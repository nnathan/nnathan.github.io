2019.11.06: An incomplete introduction to provable security

This post is primarily aimed to help uninitiated students taking a cryptography course and wondering what this provable security fuss is all about. You probably chose a course titled "Cryptography" or "Modern Cryptography" or "Applied Cryptography", and thought it had to do with learning about actual cryptographic algorithms and implementing cryptography. Sorry to say, if you are here figuring out what is going on then you were probably bamboozled. Fear not, I hope what you gte from this is the virtues of provable security and leave you a handy tip to attack typical homework exercises. If you put in the effort I believe you will come out of the course enlightened.

Provable security is a subfield of security but mostly involves cryptography. Cryptography lives in the intersection of mathematics and computer science. This distinction is important and you will find cryptography courses in math and computer science departments is wildly different and use a completely different language.

Security reductions lie at the heart of provable security. A security reduction is a "proof" stating an algorithm (or protocol) (virtually always linked to cryptography) provides a specific security guarantee. I like this excerpt from Joy of Cryptography which distills the essence of security reductions:

“any encryption scheme, when combined with this other thing in this specific way, results in a system with security property X, as long as the encryption scheme satisfies property Y”

The syntax of a security reduction is in the form of a logical implication:

[Thing-Π is Y secure] ⇒ [Thing-Π is X secure]

Thing-Π is actually a placeholder which can substituted with a real concrete implementation.

Let's look at a concrete example:

  • define Thing-Π as any encryption scheme using randomised encryption with authentication
  • replace (instance) Thing-Π with AES256-CTR-HMAC-SHA256
  • replace Y with IND-CCA2
  • replace X with IND-CPA

We have [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] ⇒ [AES256-CTR-HMAC-SHA256 is IND-CPA secure].

Note, AES256-CTR-HMAC-SHA256 is an instance of Thing-Π. We could have used something else like ChaCha20-Poly1305.

One could think of [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] as an advertisement. The "product" is the encryption scheme AES256-CTR-HMAC-SHA256 which has all the security features of IND-CCA2 security. Caveat emptor, provable security isn't a silver bullet for security, as with all matters in life, it helps to maintain a healthy level of skepticism. Provable security claims are helpful to cryptography practitioners and required by standard committees which vet and publish security schemes.

Ok, back to security reductions. Ostensibly, a security reduction communicates a set of related ideas:

  • the security of Thing-Π which is Y-secure is X-secure
  • the security of Thing-Π with Y-security reduces to X-security
  • Thing-Π with Y-security is at least as secure as X-security
  • an attack on X is an attack on Y

For example, if someone says their encryption scheme is IND-CCA2 secure then you can say it resists all attacks to IND-CPA, a weaker security notion.

As a student you will need to prove security reductions, that will make up most of your coursework. I'm going to finish with a simple tip for those who struggle with attacking a proof.

In case you aren't aware or didn't realise, most security reductions is proven using proof by contradiction. When you encounter

[Thing-Π is X-secure] ⇒ [Thing-Π is Y-secure]
. First drop Thing-Π and rewrite as [X-security] ⇔ [Y-securty]. To simplify further we reword to say [no attack against X] ⇔ [no attack against Y]. (By the way, we will bring back Thing-Π in a cool minute.) Now apply the following transforms:

  1. Apply contrapositive rule (P ⇒ Q) ⇔ (¬P ⇒ ¬Q).
    [an attack against Y] ⇒ [an attack against X]

  2. Apply material implication (P ⇒ Q) ⇔ (P∨¬Q):

    [an attack against Y] or [no attack against X]
  3. Negate statement using De-Morgan's law ¬(P ∨ Q) ⇔ (¬P ∧ ¬Q) [no attack against Y]
    4. ∧ [and attack against X]

This probably sounds wonderful or pointless theory. At the end of the day you want to know "is the thing secure?" The provable security framework helps to answer the question by defining security and then prove it. Some consider it a gold standard. Students who put in the effort will probably come out feeling enlightened and come to the same conclusion. Others believe that provable security is mental masturbation with harmful side-effects. And the foremost experts refuse to allow critique of their dogma. Then there are the practitioners, also called applied cryptographers. Some don't bother advertise using provable security jargon. Personally I view provable security as a decent benchmark that cryptographers should adhere to, but remains insufficient as a measure of security.

So far I have given a superficial view to provable security. I didn't even touch on assumptions which I consider the dark side of provable security, and may be a subject of a later post. I hope in the course of your studies, this inchoate introduction can will help fill in missing context. If none of it makes sense now, don't think too hard about it, and prove everything you can.
