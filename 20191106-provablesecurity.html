2019.11.06: A brief introduction to provable security

This post is primarily aimed to help uninitiated students taking a cryptography course and wondering what this provable security stuff is all about. Sorry to say, you have probably been bamboozled. In any case, I hope to explain the virtues of provable security and give newcomers a simple tip for their homework problems.

Provable security is a subfield of security but mostly involves cryptography. Cryptography lives in the intersection of mathematics and computer science. This distinction is important.

Security reductions lie at the heart of provable security. A security reduction is a "proof" stating an algorithm (usually involving cryptographic) makes a security claim. I like this excerpt from The Joy of Cryptography to describe the essence of security reductions:

“any encryption scheme, when combined with this other thing in this specific way, results in a system with security property X, as long as the encryption scheme satisfies property Y”

The syntax of a security reduction is in the form of a logical implication:

[Thing-Π is Y secure] ⇒ [Thing-Π is X secure]

Thing-Π is actually a placeholder which can substituted with a real concrete implementation.

Let's look at a concrete example:

  • define Thing-Π as any encryption scheme using randomised encryption with authentication
  • replace (instance) Thing-Π with AES256-CTR-HMAC-SHA256
  • replace Y with IND-CCA2
  • replace X with IND-CPA

We have [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] ⇒ [AES256-CTR-HMAC-SHA256 is IND-CPA secure].

Note, AES256-CTR-HMAC-SHA256 is an instance of Thing-Π. We could have used something else like ChaCha20-Poly1305.

One can think of [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] as an advertisement. The "product" is the encryption scheme AES256-CTR-HMAC-SHA256 which has all the security features of IND-CCA2 security. Caveat emptor, provable security isn't a silver bullet for security, as with all matters in life, it helps to maintain a healthy level of skepticism. Provable security claims are helpful to cryptography practitioners and required by standard committees which vet and publish security schemes.

This probably sounds wonderful or pointless theory. At the end of the day you want to know "is the thing secure?" The provable security framework helps to answer the question by attempting to define security and then prove it. Some consider it a gold standard. Students who put in the effort will probably come out feeling enlightened and come to the same conclusion. Others believe that provable security is mental masturbation. Some practitioners don't bother advertising using provable security jargon. Personally I think provable security sets a decent benchmark for applied cryptographers but is incomplete in defining security, and provies a limited guarantee.

Ok, back to security reductions. Ostensibly, a security reduction communicates a set of related ideas:

  • the security of Thing-Π which is Y-secure is X-secure
  • the security of Thing-Π with Y-security reduces to X-security
  • Thing-Π with Y-security is at least as secure as X-security
  • an attack on X is an attack on Y.
For example, if someone says their encryption scheme is IND-CCA2 secure then you can say it resists all attacks to IND-CPA, a weaker security property.

There is so much more to say about provable security. I haven't even touched on assumptions which is what I consider the dark side of provable security. But I hope this is sufficient to mull over in the meantime. It is my hope, during the course of your studies, this inchoate introduction can help fill in some missing context. If none of this makes sense, mull over it.

So I studied provable security and I wanted to provide a tip which is stupidly simple but I find helpful to the uninitiated.

Incase you weren't aware or didn't realise, most security reductions are proven using proof by contradiction. When you encounter

[Thing-Π is X-secure] ⇒ [Thing-Π is Y-secure]
do the following:
  1. rewrite in contrapositive: [Thing-Π is not Y-secure] ⇒ [Thing-Π is not X-secure]
  2. convert to material implication: [Thing-Π is Y-secure] or [Thing-Π is not X-secure]
  3. apply De-Morgan's law to negate the statement: [Thing-Π is Y-secure] and [Thing-Π is not X-secure]
