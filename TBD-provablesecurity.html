TBD: An incomplete introduction to provable security

This post is primarily aimed to help uninitiated students taking a cryptography course and wondering what this provable security fuss is all about. You probably chose a course titled "Cryptography" or "Modern Cryptography" or "Applied Cryptography", and thought it had to do with learning about actual cryptographic algorithms and implementing cryptography. Sorry to say, if you are here trying to figure out what the hell is going on and start to think cryptography is all about proofs then you were probably bamboozled. Fear not, this post is to help fill in the gaps on the why of provable security and what value it provides.

Provable security is a subfield of security which almost always combines cryptography in some way. Cryptography lives in the intersection of mathematics and computer science. This distinction is important and you will find cryptography courses in math and computer science departments is wildly different and use a completely different language.

Security reductions lie at the heart of provable security. A security reduction is a "proof" stating an algorithm somehow related to cryptography provides a specific security guarantee. I like this excerpt from Joy of Cryptography which distills the essence of security reductions:

“any encryption scheme, when combined with this other thing in this specific way, results in a system with security property X, as long as the encryption scheme satisfies property Y”

The syntax of a security reduction is in the form of a logical implication:

[Thing-Π is Y secure] ⇒ [Thing-Π is X secure]

Thing-Π is actually a placeholder which can substituted with a real concrete implementation.

Let's look at a concrete example:

  • define Thing-Π as any encryption scheme using randomised encryption with authentication
  • replace (instance) Thing-Π with AES256-CTR-HMAC-SHA256
  • replace Y with IND-CCA2
  • replace X with IND-CPA

We have [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] ⇒ [AES256-CTR-HMAC-SHA256 is IND-CPA secure]. This is an actual security claim that we believe is true. But the details like AES256-CTR-HMAC-SHA256 and IND-{CPA,CCA2} is irrelevant. From experience, I never heard the words AES or SHA256 uttered in any provable security course that I took.

Note, AES256-CTR-HMAC-SHA256 is an instance of Thing-Π. We could have used something else like ChaCha20-Poly1305.

One could think of [AES256-CTR-HMAC-SHA256 is IND-CCA2 secure] as an advertisement. The "product" is the encryption scheme AES256-CTR-HMAC-SHA256 which has all the security features of IND-CCA2 security. Caveat emptor, provable security isn't a silver bullet for security, as with all matters in life, it helps to maintain a healthy level of skepticism. Provable security claims are helpful to cryptography practitioners and required by standard committees which vet and publish security schemes.

Ostensibly, a security reduction communicates a set of related ideas:

  • the security of Thing-Π which is Y-secure is X-secure
  • the security of Thing-Π with Y-security reduces to X-security
  • Thing-Π with Y-security is at least as secure as X-security
  • an attack on X is an attack on Y (note to self: while this is usually the case - I don't think you can derive a generic claim in all instances)

For example, if someone says their encryption scheme is IND-CCA2 secure then you can say it resists all attacks to IND-CPA, a weaker security notion.

As a student you will need to prove security reductions, which makes up most of your coursework. In what follows I will show how to attack a securiy reduction.

Showing a security reduction

In case you aren't aware or didn't realise, we prove most security reductions using proof by contradiction.

When you encounter [Thing-Π is X-secure] ⇒ [Thing-Π is Y-secure] drop Thing-Π so we have [X-security] ⇔ [Y-security]. Don't worry, we will bring back Thing-Π in a cool minute.

To simplify further we reword to say [no attack against X] ⇔ [no attack against Y]. An attack is some violation of X or Y. Now apply the transforms:

  1. Contrapositive rule (P ⇒ Q) ⇔ (¬P ⇒ ¬Q):
    [an attack against Y] ⇒ [an attack against X]
  2. Material implication (P ⇒ Q) ⇔ (P ∨ ¬Q):
    [an attack against Y] or [no attack against X]
  3. Negate statement using De-Morgan's law ¬(P ∨ Q) ⇔ (¬P ∧ ¬Q):
    [no attack against Y] ∧ [an attack against X]

Here's a handy tip in life: when you prove implication P⇒Q using contradiction you should have on instant recall: P ∧ ¬Q. That is, assume P to be true, using ¬Q show P is false, (the contradiction) which states P∧¬Q is false, and everything else is true. The setup for contradiction is easy. The hard part is showing the contradiction.

Back to the proof. We have [no attack against Y] ∧ [an attack against X]. Let's put Thing-Π back. Now we have [no attack against Y for Thing-Π] ∧ [an attack against X for Thing-Π]. Now this is the detail you need to fill in: you need to show an attack on X for Thing-Π can convert into an attack on Y. Then we contradict the assumption [no attack against Y for Thing-Π]. Which means the entire statement is false. And this means the negation (3) is true. This truth propagates backwards to the original statement [Thing-Π is X-secure] ⇒ [Thing-Π is Y-secure], which we conclude is true.

Note that attacking the proof directly can be futile. That is you first show the pair [Thing-Π is X-secure] and [Thing-Π is not Y-secure] is false, and show all remaining cases true.

"Why do I care?"

This probably sounds wonderful or pointless theory. At the end of the day you want to know "is the thing secure?" The provable security framework helps to answer the question by defining security and then prove it. Some consider it a gold standard. Students who put in the effort will probably come out feeling (potentially a false sense of) enlightened and come to the same conclusion. There is a divide among cryptographers on the value of provable security. It all started with a critique created a major upset among the foremost experts demanding a retraction to what they consider a threat to their existence. Computer scientists sided with provable security and mathematicians went their own way. Then there are the practitioners who develop and provide cryptography solutions in the hand of users. I like to call these people applied cryptographers. One applied cryptographer I know of doesn't bother advertising using provable security jargon. Another wrote an encyclopedia which was used throughout the 90s but is outdated and no longer in fashion. In my own observations I have noticed in recent years some cryptography libraries now give a nod to provable security, e.g. nacl, My advice for now, is to know opinions differ on the value of provable security, but the best way to understand the arguments made, is to first understand the machinery of provable security. So don't think that people critiquing provable security is not a reason to not do your homework. And with all matters in life, it is helpful to maintain healthy skepticism.

Insofar I have given a superficial view to provable security. I omitted a lot of detail. I didn't even touch on assumptions which I consider as the dark side provable security, and will be a subject of a later post.

I hope in the course of your studies, this inchoate introduction helps elucidate the gaps as to the why of provable security, and the general concensus on applicability. If none of it makes sense now, don't think too hard about it, and prove everything you can.
